## Overview

Knowledge bases let your agent **retrieve answers from your content** — product docs, FAQs, policies, and files.

**Flow:** Create KB → add sources (async) → poll jobs → attach `kb_id` to LLM model → agent uses it on calls.

---

## All endpoints

| Step | Method | Endpoint |
| --- | --- | --- |
| 1 | `POST` | `/v1/knowledge-bases` |
| 2 | `POST` | `/v1/knowledge-bases/{kb_id}/sources` |
| 3 | `GET` | `/v1/knowledge-bases/{kb_id}/jobs/{job_id}` |
| 4 | `PATCH` | `/v1/models/{llm_id}` |
| — | `GET` | `/v1/knowledge-bases/{kb_id}/sources` |
| — | `GET` | `/v1/knowledge-bases/{kb_id}/jobs` |

## How retrieval works

OneInbox chooses a mode based on total KB token budget (default **10k tokens**):

- **≤ 10k tokens** — content is injected into the LLM system prompt at call time (no extra retrieval round-trips)
- **> 10k tokens** — vector retrieval with a synthetic `search_knowledge_base` tool the LLM can call

Source uploads return **`202 Accepted`** immediately. Poll the **`job_id`** until status is `completed`.

---

<Steps>

<Step title="1. Create a knowledge base">

```bash
curl -X POST http://13.207.23.185:8000/v1/knowledge-bases \
  -H "Authorization: Bearer <api_key>" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Product Docs",
    "chunk_size": 512,
    "multilingual": false
  }'
```


</Step>

<Step title="2a. Add a file source">

Upload PDF, DOCX, XLSX, TXT, or MD (max **50 MiB** per file). Use `multipart/form-data`:

```bash
curl -X POST "http://13.207.23.185:8000/v1/knowledge-bases/<kb_id>/sources" \
  -H "Authorization: Bearer <api_key>" \
  -F "file=@/path/to/document.pdf"
```

</Step>

<Step title="2b. Add a URL source">

```bash
curl -X POST "http://13.207.23.185:8000/v1/knowledge-bases/<kb_id>/sources" \
  -H "Authorization: Bearer <api_key>" \
  -F "type=url" \
  -F "source=https://docs.example.com/faq"
```

</Step>

<Step title="2c. Add inline text">

```bash
curl -X POST "http://13.207.23.185:8000/v1/knowledge-bases/<kb_id>/sources" \
  -H "Authorization: Bearer <api_key>" \
  -F "type=text" \
  -F "source=Our return policy is 30 days, no questions asked."
```

</Step>

<Step title="3. Poll the indexing job">

All source uploads return **202** with a job ID:

<ResponseExample>

```json
{
  "job": {
    "id": "job_abc123",
    "status": "queued"
  }
}
```

</ResponseExample>

Poll until `completed` or `failed`:

```bash
curl "http://13.207.23.185:8000/v1/knowledge-bases/<kb_id>/jobs/<job_id>" \
  -H "Authorization: Bearer <api_key>"
```

Status flow: `queued` → `running` → `completed` | `failed`. On success, `payload` includes `tokens`, `chunks`, and `vector_indexed`.

List recent jobs:

```bash
curl "http://13.207.23.185:8000/v1/knowledge-bases/<kb_id>/jobs" \
  -H "Authorization: Bearer <api_key>"
```

</Step>

<Step title="4. List sources (optional)">

```bash
curl "http://13.207.23.185:8000/v1/knowledge-bases/<kb_id>/sources" \
  -H "Authorization: Bearer <api_key>"
```

Source rows include `status` (`processing` / `ready` / `error`), `file_name`, `token_count`, `chunk_count`, and `error_message` when applicable.

</Step>

<Step title="5. Attach to your LLM model">

```bash
curl -X PATCH "http://13.207.23.185:8000/v1/models/<llm_id>" \
  -H "Authorization: Bearer <api_key>" \
  -H "Content-Type: application/json" \
  -d '{ "knowledge_base_ids": ["<kb_id>"] }'
```

Agents using this `llm_id` will use the KB on the next call.

</Step>

</Steps>

## Manage knowledge bases

```bash
curl http://13.207.23.185:8000/v1/knowledge-bases \
  -H "Authorization: Bearer <api_key>"

curl -X DELETE "http://13.207.23.185:8000/v1/knowledge-bases/<kb_id>" \
  -H "Authorization: Bearer <api_key>"
```
